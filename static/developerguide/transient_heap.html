Transient Heap¶

Reference: https://wiki.corp.adobe.com/display/t4/transient_heap

transient_heap is a heterogeneous container class capable of containing instances of arbitrary C++ data types. transient_heap makes allocations in large chunks and uses that memory to compactly suballocate the objects it contains. One important difference is that the objects can’t be individually deallocated; they are destroyed in batches by transient_heap’s destructor, or, less frequently, a utility class called transient_snapshot. Objects are destroyed in LIFO order. Internally, transient_heap maintains a linked list of objects it has allocated so that it knows what to destroy when it goes out of scope, but it only bothers to add objects with non-trivial destructors to this list.

If you design the data structures you allocate out of the transient_heap carefully, you can store extremely complex graphs of objects without incurring the overhead of complex destruction semantics, and your code should also benefit from memory collocation. For example, it would be very easy to design a tree data structure whose edge nodes are all raw pointers allocated out of the same heap. If all the values stored in the nodes have trivial destructors as well, the cost of deleting the structure is simply the cost of deallocating each of the large chunks of memory. Traversal of the tree will be faster as the nodes will be physically collocated in the transient_heap’s chunks.

The tradeoff, of course, is that this data structure is grow-only; while you can delete nodes you can’t reclaim the memory used by those nodes without destroying the entire heap.

Quick Start¶

Declare a transient_heap on the stack:

using namespace t4;

void main(void)
{
        transient_heap my_heap;

Allocate an instance of ogre using memory in my_heap:

// Create an ogre with 50 marbles.  This invokes ogre's ctor, passing 50 as a parameter
        ogre* my_ogre = make_transient<ogre>(my_heap, 50);

Allocate an array of 100 integers using my_heap:

// Create an array of 100 ints.  Because ints are trivially destructible,
// there's no overhead to this beyond the memory occupied by the ints themselves.
int* my_ints = make_transient_array<int>(my_heap, 100);

Create a vector<int> using memory stored in a transient_heap :

// my_int_vec is a raw pointer to a vector
        auto my_int_vec = make_transient<std::vector<int, transient_allocator<int>>>(my_heap);
        my_int_vec->push_back(23);

// "transient_vector" is just a typedef alias to std::vector<int, transient_allocator<T>>.
// This is usage is identical to, but preferred over, the above
        auto my_int_vec2 = make_transient<t4::transient_vector<int>>(my_heap);
        my_int_vec2->push_back(23);

Warning

Be careful when using t4::transient_vector. transient_allocator can’t free memory, so if a transient_vector needs to resize (because, for example, push_back is called on a vector whose size equals its capacity) the bytes in the original allocation will be wasted. If you don’t know the number of elements in advance, you probably shouldn’t be using a transient_vector. If you do know the number of elements in advance, use reserve() to make space for them in advance.

t4::chunked_vector is a good alternative to vector if you don’t know the number of elements in advance.

Deallocation of the objects allocated in my_heap:

}

Note

You can’t destroy objects allocated out of a transient_heap! At least not individually. Objects are destroyed in bulk when the transient_heap is destroyed. (See also transient_snapshot, below).

transient_heap Applications¶

transient_heap has two main categories of use. One is for building complex graphs of objects where you don’t ever intend to delete objects, and the other is for quickly allocating objects that are used in the execution of a single algorithm, and are only needed for the lifetime of the algorithm’s execution.

Grow-Only or Immutable Graphs of Heterogeneous Objects¶

Unlike other container classes, transient_heap can contain arbitrary objects, and store them efficiently–if an object has a trivial destructor there is no overhead apart from storage of the object itself. The DisplayList class, which is used to contain all the objects in a single page of PDF, is a great example of this. When a DisplayList is parsed, all of the objects are allocated out of a single transient_heap. DisplayLists contains a huge number of different kinds of objects representing text, paths, color values, clips, transparency groups, etc., and those objects have a hierarchical relation to each other.

In a read-only environment transient_heap is ideal for DisplayList – the data structure is built at parse time, and objects are allocated out of the heap in roughly the same order they will be used, so we benefit from the collocation in the transient heap’s blocks of memory. As we start editing, though, some of this benefit goes away–there’s no way to delete an individual object in a transient_heap (it can be disconnected from the graph, sure, but it will occupy memory for the lifetime of the heap).

We also use transient_heap in this way for other data structures that are grow-only (or immutable), and have the potential to be very complex. For example, the Structure class uses a single transient_heap to store a single page’s inferred structure. Some CharCodeMaps, which are used to convert sequences of bytes into character codes, also use this technique.

Algorithm-Lifetime Bound Transient Allocations¶

Tetraphilia provides an object called the “thread-specific transient heap” which attaches a single instance of transient_heap to threads that use the core. We use this to support algorithms that want to be able to create objects of arbitrary complexity while still enabling the common, simple cases to proceed with minimal overhead.

An example: a single execution of the algorithm by which an object in a PDF contents stream is painted into a pixel buffer can be very simple (e.g., paint a bezier path with a solid color fill) or complex (e.g., paint the results of this transparency group into this knockout group using the multiply blend mode and a soft mask). The GetMaskRasterPainter function takes two input RasterPainters–the source and the backdrop–and returns a pointer to a RasterPainter representing the interpolation of the source and the backdrop based on the mask as described by the gstate.

The thread-specific transient heap allows us to return a pointer without the caller having to manage it. If the gstate contains “interesting” mask alpha, GetMaskRasterPainter returns an object allocated out of that heap that knows how to perform the appropriate linear interpolation between the two inputs. If there is no interesting alpha, GetMaskRasterPainter simply returns the pointer to the source input. In either case the caller doesn’t have to explicitly dispose of the returned pointer.

The thread-specific transient heap is retrieved using the get_TransientHeap function:

auto& tls_heap = get_TransientHeap<app_traits>();

// This variant also exists, and does exactly the same thing
auto& tls_heap2 = get_TransientHeap(context);

As of this writing a de-traitsified version of this function (which returns a pointer) is available in the multithreading branch:

auto& tls_heap = *t4::get_thread_heap();

Destruction of Objects in a transient_heap¶

C++ objects allocated inside a transient_heap are not destroyed, and their pointers not deallocated, until the transient_heap itself is destroyed (unless you’re using transient_snapshots, described below):

struct noisy
{
        static int s_count = 0;
        int m_count = s_count++; // not threadsafe, but this is just an example

        noisy(void) { std::cout << "creating " << m_count << std::endl; }
        ~noisy(void) { std::cout << "destroying " << m_count << std::endl; }
};

std::cout << "start" << std::endl;
{
        transient_heap my_heap;
        noisy* a = make_transient<noisy>();
        std::cout << "A" << std::endl;
        noisy* b = make_transient<noisy>();
        std::cout << "B" << std::endl;
}
std::cout << "stop" << std::endl;

The above program outputs:

start
creating noisy 0
A
creating noisy 1
B
destroying noisy 1
destroying noisy 0
stop

Snapshots¶

The transient_snapshot class can be used to free subsets of a transient_heap’s data. This technique is typically used only with the “Algorithm-Specific Transient Allocations” use case, described above. A transient_snapshot is a scoped entity that is instantiated on a reference to a transient_heap. Its destructor restores the transient_heap’s state to its state at the time the transient_snapshot’s constructor is executed. An example:

std::cout << "start" << std::endl;
{
        transient_heap my_heap;
        noisy* a = make_transient<noisy>();
        std::cout << "A" << std::endl;
        {
                transient_snapshot ss(my_heap);
                noisy* b = make_transient<noisy>();
                std::cout << "B" << std::endl;
                {
                        transient_snapshot ss2(my_heap);
                        noisy* c = make_transient<noisy>();
                        std::cout << "C" << std::endl;
                }
                std::cout << "D" << std::endl;
        }
        std::cout << "E" << std::endl;
}
std::cout << "stop" << std::endl;

The above program outputs:

start
creating noisy 0
A
creating noisy 1
B
creating noisy 2
C
destroying noisy 2
D
destroying noisy 1
E
destroying noisy 0
stop

Multiple transient_snapshots may exist on the same transient_heap, but they must nest. Objects are destroyed in LIFO order.
