T4 File Organization¶

T4 copies the approach used by the Boost libraries to organize header and source files.

About Header Files¶

There are three distinct clients of header files, leading to three distinct categories:

  1. Public. The client of the interface exposed by the header files should be able to get their work done seeing only these files. In theory, they are only interested in learning and using the documented, public interfaces.

  2. Detail. The client’s compiler needs to see these. This is modern C++; a great deal of what we consider “implementation details” must nevertheless be visible to the compiler.

  3. Implementation. T4’s implementation needs to “see” these files, but they are used only by .cpp’s and don’t expose any data structures required by either of the two aforesaid categories.

Where Everything Goes¶

Imagine you’re writing a new class for public consumption named sprocket. The first thing clients need to know, of course, is what header file to include. By perfectly obvious convention, clients should get full access to the sprocket class with the following #include:

#include "t4/sprocket.hpp"

Back in the olden days we’d just have that one file, and we’d write a corresponding sprocket.cpp. We’d tell the client everything they need to know is in sprocket.hpp and we’d declare sprocket.cpp to be a bunch of implementation that the caller shouldn’t have to know about to use a sprocket. But modern C++, for better or for worse, has to put a great deal of implementation in header files which must be visible to the compiler. Furthermore, good class definitions often require supporting classes and functions which aren’t part of the client interface but must nevertheless be visible (and, in the case of methods, even public). The goal is that t4/sprocket.hpp should contain:

  1. Declarations of public functions

  2. Definitions of public classes

  3. Declarations of the methods of public classes

  4. Documentation of same

  5. Not much else

Everything else should go in other header files, depending on their nature. Here’s how we might break down a particularly complex sprocket implementation:

detail/sprocket.hpp

/*
        detail/sprocket.hpp
        Copyright 2016 Adobe Systems Inc., All Rights Reserved, etc., etc.

*/

#ifndef T4_DETAIL_SPROCKET_HPP_
#define T4_DETAIL_SPROCKET_HPP_

namespace t4 {
namespace sprocket_detail {

class impl
{
public:
        size_t get_widget_count( void ) const { return get_widget_count_impl(); }
        boost::any_iterator<widget, std::input_iterator_tag> begin( void ) const { return begin_impl(); }
        boost::any_iterator<widget, std::input_iterator_tag> end( void ) const { return end_impl(); }

private:
        virtual size_t get_widget_count_impl( void ) const = 0;
        virtual boost::any_iterator<widget, std::input_iterator_tag> begin_impl( void ) const = 0;
        virtual boost::any_iterator<widget, std::input_iterator_tag> end_impl( void ) const = 0;
};

} // end namespace sprocket_detail
} // end namespace t4

#endif /* T4_DETAIL_SPROCKET_HPP_ */

sprocket.hpp

/*
        sprocket.hpp

        Copyright 2016 Adobe Systems Inc., All Rights Reserved, etc., etc.

*/

#ifndef T4_SPROCKET_HPP_
#define T4_SPROCKET_HPP_

#include "t4/detail/sprocket.hpp"
#include "t4/widget.hpp"

namespace t4 {

class sprocket
{
        shared_ptr<sprocket_detail::impl> m_impl;

public:
        sprocket( void ) {}
        inline sprocket( shared_ptr<sprocket_detail::impl> impl );
        inline sprocket& operator=( sprocket rhs );

        /*
                Get number of widgets in this sprocket.
        */
        size_t get_widget_count( void ) const;

        /*
                Call f on each widget in the sprocket. The following expression must be valid:
                        f(w)
                Where w is a const reference to a widget.
        */
        template<typename F>
        void for_each_widget( F&& f ) const;

};

} // end namespace t4

#include "t4/detail/sprocket_impl.hpp"

#endif /* T4_SPROCKET_HPP_ */

detail/sprocket_impl.hpp

/*

        detail/sprocket_impl.hpp

        Copyright 2016 Adobe Systems Inc., All Rights Reserved, etc., etc.

*/

#ifndef T4_DETAIL_SPROCKET_IMPL_HPP_
#define T4_DETAIL_SPROCKET_IMPL_HPP_

namespace t4 {

inline sprocket::sprocket( shared_ptr<sprocket_detail::impl> impl )
        : m_impl( std::move( impl ) )
{}

inline sprocket& sprocket::operator=( sprocket rhs )
{
        m_impl = std::move( rhs );
        return *this;
}

template<typename F>
void sprocket::for_each_widget( F&& f ) const
{
        std::for_each( m_impl->begin(), m_impl->end(), std::move( f ) );
}

} // end namespace t4

#endif /* T4_DETAIL_SPROCKET_IMPL_HPP_ */

sprocket.cpp

/*
        libs/sprocket.cpp

        Copyright 2016 Adobe Systems Inc., All Rights Reserved, etc., etc.

*/

#include "t4/sprocket.hpp"

namespace t4 {

size_t sprocket::get_widget_count( void ) const
{
        return m_impl->get_widget_count();
}

} // end namespace t4

detail/sprocket.hpp should contain:

  1. Forward declarations

  2. “Detail” declarations, and definitions if useful

detail/sprocket_impl.hpp should contain:

  1. Template and inline method/function definitions

  2. Any classes/functions/template needed to support #1 (these will probably belong in a detail namespace)

libs/sprocket.cpp should contain:

  1. Definitions of non-inline, non-template functions and methods

About Detail Namespaces¶

We use “detail” namespaces to define classes, functions and typedefs that need to be available to the compiler when compiling client code, but aren’t technically part of our public API and we want it to be clear to clients that they shouldn’t directly use them. A typical detail namespace will take the form xxx_detail, where xxx is replaced by the particular subset of t4 functionality being supported. For example, t4::malloc is in the t4 namespace, but memory_context, a “private” class that must nevertheless be visible at compile time, is in t4::memory_detail. It’s reasonable to ask why we break down detail namespaces like this if we’re not breaking up the parent t4 namespace. That’s just how I decided to do it, reasoning that it’s easier, and worth the trouble, to manage identifier collisions in the client-facing t4 namespace, and much more difficult to manage identifier collisions across the implementation support code for those public identifiers.

Implementation Headers¶

If for whatever reason your .cpp’s are growing in complexity and they need to share declarations that don’t need to be visible to either the client or the client’s compiler, you are free to put .hpp’s next to the .cpp’s in the t4/libs directory.

How Far to Go With This¶

Don’t do this any more than is strictly necessary. Sometimes you can code up the whole class in a single .hpp and that’s fine. The goal here is to provide a reasonably uncluttered view of the public interface. There’s nothing wrong with a simple inline method definition, e.g.:

class foo
{
        ...
public:
        size_t size( void ) const { return m_elements.size(); }
}

Even larger method bodies are okay, especially if you think their implementation might actually help a client understand a class better. But if it starts to get too large (10 lines? Use your judgement), please consider moving the method implementation to detail/sprocket_impl.hpp (for template methods and functions) or libs/sprocket.cpp (for non-template methods and functions).

Going Even Farther¶

For modules with extensive interdependencies, it can be necessary to separate the client API even further. Sometimes you’ll need to use the following patterns:

  • sprocket_fwd.hpp for forward declarations of templates, classes and functions that are part of the public API

  • sprocket_def.hpp for definitions of classes that are part of the public API without the accompanying #include "detail/sprocket_impl.hpp" statement. This can be necessary if you’re writing two mutually referential classes with method implementations that need access to both class definitions.
